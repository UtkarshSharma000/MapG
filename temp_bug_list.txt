================================================================================
ODYSSEY INTERPLANETARY FLIGHT ENGINE - DETAILED BUG SCAN REPORT
================================================================================

This document contains the complete and structured list of bugs, mathematical mismatches, and UI gaps identified across the Odyssey codebase.

--------------------------------------------------------------------------------
1. FILE: /src/TrajectoryOptimizer.tsx (Porkchop & Sequence Optimizations)
--------------------------------------------------------------------------------
- [BUG 1.1] "Missing Renderer": The TrajectoryOptimizer is imported in App.tsx but is completely omitted from the jsx markup. The user cannot access the multi-leg or porkchop trajectory planning component anywhere on the overlay panels.
- [BUG 1.2] "Planet Lookups & ID Offset Safety": The mapping of PLANETS uses 1-based ids (1: Mercury, 2: Venus, 3: Earth, 4: Mars, 5: Jupiter, 6: Saturn) inside TrajectoryOptimizer, while the visualizer's elements sometimes map properties on 0-based arrays or different schemas. We need defensive checks to ensure bounds of [1-8] are fully honored when translating leg IDs, returning Mercury-Neptune names.

--------------------------------------------------------------------------------
2. FILE: /src/physics.ts & /src/workers/trajectory.worker.ts (Orbital Physics)
--------------------------------------------------------------------------------
- [BUG 2.1] "Inconsistent Astronomical Constants": Constant drift exists across physics.ts, the web worker, and the Python FastAPI bridge.
    * src/physics.ts utilizes GM_SUN derived as G * M_SUN (1.327518568e20 m^3/s^2),
    * local_backend/main.py uses MU_SUN = 1.32712440018e20,
    * workers/trajectory.worker.ts uses GM_SUN = 1.327124e20.
  This creates visible coordinate offsets and misalignments between predicted paths and real-time visual vectors in ThreeJS.
- [BUG 2.2] "Symbol Uniformity": Standardize the constant names to MU_SUN and align the gravitational value precisely to 1.32712440018e20 across the primary modules.

--------------------------------------------------------------------------------
3. FILE: /src/App.tsx (Main Integration & State Managers)
--------------------------------------------------------------------------------
- [BUG 3.1] "Trajectory Panel Incorporation": Render <TrajectoryOptimizer> in the main flight overlays when `showMissionPanel` is enabled and `isSimulatorRunning` is active.
- [BUG 3.2] "HandleApply ID Mappings": When applying optimal transfer trajectories via `handleApply`, map incoming leg structures and planet indices correctly from 1-based dictionary identifiers used by the optimizer back to the OrbitSimulator's expected configurations without numerical shifting.

--------------------------------------------------------------------------------
4. FILE: /local_backend/main.py (FastAPI Python Bridge)
--------------------------------------------------------------------------------
- [BUG 4.1] "Venus/Mars to Earth Return plotting failure": In `trajectory_preview` (lines 348-349), the check is:
  `if target_planet and target_planet != "Earth":`
  When planning a Venus/Mars return to Earth, `target_planet` is "Earth". Thus, the endpoint skips the heliocentric interplanetary Hohmann path rendering and falls back to Earth LEO launch rendering lines (which renders a tiny orbital circle near the solar center). This is a critical orbital plotting bug!
  *Fix*: Adjust check to support heliocentric paths whenever launch planet differs from target planet.

--------------------------------------------------------------------------------
5. FILE: /src/components/TelemetryPanel.tsx (Telemetry Draggable Frame)
--------------------------------------------------------------------------------
- [BUG 5.1] "Server booting Connection Refusals": When the uvicorn socket (port 8000) is starting up, fetch errors flood the browser console with uncaught connection warnings. Include a graceful fallback check inside the interval fetches.
